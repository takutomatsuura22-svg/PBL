# 知識調査コマンド

## Description
指定されたテーマについて網羅的に知識を調査し、詳細なノートを複数作成します。

## Prompt
以下のテーマについて、網羅的に詳細な知識を調査して、01_Inboxフォルダに複数のMarkdownファイルを作成してください。

**重要**: 1項目1ファイルの原則を厳守してください。機能単位、概念単位で必ず別々のファイルを作成します。

引数:
- テーマ: 調査したい技術や概念（例: "React Hooks", "TypeScript", "PostgreSQL"）

要件:
1. 調査項目を洗い出す
   - 基本概念
   - 主要機能（機能ごとに別ファイル）
   - 実装方法
   - ベストプラクティス
   - よくある問題
   - パフォーマンス
   - セキュリティ
   - 関連技術との比較

2. **必須：細分化の徹底**
   
   **重要ルール**: 1つの知識項目 = 1つのファイル
   
   - ✅ 良い例（React Hooksの場合）:
     - `YYYYMMDD_React_useStateの使い方.md`
     - `YYYYMMDD_React_useEffectの使い方.md`
     - `YYYYMMDD_React_useContextの使い方.md`
     - `YYYYMMDD_React_Custom Hooksの作成.md`
     - `YYYYMMDD_React_Hooksのルールと注意点.md`
   
   - ❌ 悪い例:
     - `YYYYMMDD_React Hooksまとめ.md`（全てを1ファイルに）
   
   細分化の判断基準:
   - 1つのトピックで独立したプロジェクトが作れるか？
   - 別のトピックと混ぜても理解に支障がないか？
   - 検索した時に「○○について知りたい」で見つかるか？
   - 10分以上説明する内容があるか？
   - その機能・概念だけで完結した説明ができるか？

3. 各ノートの作成
   - ファイル名: `YYYYMMDD_[テーマ]_[サブトピック].md`（日本語推奨）
   - 内容: 詳細な説明、コード例、使用例を含む
   - リンク: 関連ノートと必ずリンク
   - タグ: 適切なタグを付与

4. ノート構成
```markdown
# [トピック名]

## 概要
[1-2文で要約]

## 詳細
[詳細な説明 - 実践的に使える内容]

## 基本的な使い方
[コード例を含む基本使い方]

## 高度な使い方
[より高度なテクニック]

## コード例
```tsx
// 実践的なコード例
```

## メリット・デメリット
- ✅ メリット1
- ✅ メリット2
- ❌ デメリット1
- ❌ デメリット2

## 使用例
[実際の使用シナリオ]

## よくある問題と解決策
- 問題1: [説明]
  解決策: [方法]

## ベストプラクティス
1. [推奨事項1]
2. [推奨事項2]

## 関連技術との比較
| 技術 | 特徴 | 適用場面 |
|------|------|----------|
| A | ... | ... |
| B | ... | ... |

## 関連ノート
- [[関連ノート1]]
- [[関連ノート2]]

#テーマ #サブトピック #memory候補
```

5. 調査の深さ
   - レベル3: 実践的な理解を目指す
   - 「何か」「どう使うか」「なぜそれを使うか」を理解
   - 実践的なコード例を含む
   - ベストプラクティスを含む

6. **ファイル作成ルール（必須）**
   
   **絶対に守る原則**: 1機能1ファイル、1概念1ファイル
   
   - 今日の日付プレフィックス: YYYYMMDD_
   - 日本語ファイル名を推奨
   - **1ファイル1機能・1概念のみ**（複数の機能を1ファイルにまとめない）
   - 詳細に書く（表面的な説明は避ける）
   - ファイル名は「○○の△△」形式で具体的に（例: `useStateの使い方.md`）
   
   ファイル命名例:
   - テーマ: Next.jsの場合
     - `YYYYMMDD_Next.jsとは何か.md`
     - `YYYYMMDD_Next.jsのApp Router.md`
     - `YYYYMMDD_Next.jsのServer Components.md`
     - `YYYYMMDD_Next.jsのClient Components.md`
     - `YYYYMMDD_Next.jsのルーティング.md`
     - `YYYYMMDD_Next.jsのデータフェッチング.md`

手順:
1. 指定されたテーマを分析
2. **テーマを細分化**
   - 機能リスト、概念リストを作成
   - 各機能・概念を個別のファイルに分割する計画を立てる
   - 最低5ファイル以上（通常は10-20ファイル）を作成
3. **各サブトピックについて詳細に調査**
   - 1ファイル = 1機能・1概念のみ
   - それぞれ独立して完結した内容にする
4. **各ファイルを01_Inboxフォルダに作成**
   - ファイル名は具体的に（機能名・概念名を含める）
   - 内容は詳細に（コード例、使用例を含む）
   - 日付プレフィックス: YYYYMMDD_
5. **ファイル間にリンクを設定**
   - 関連するノート同士をリンクで繋ぐ
   - 親概念→子概念の構造を意識
6. **概要レポートを作成**
   - 作成したファイル一覧
   - 調査範囲のチェック
7. **後で整理することを推奨**
   - 週次レビューで `04_Memory/` または `03_Input/` に振り分け

出力形式:
```markdown
## 調査完了レポート

### 作成したノート: X個

1. YYYYMMDD_[テーマ]_[サブトピック1].md
   - 内容: [概要]
   
2. YYYYMMDD_[テーマ]_[サブトピック2].md
   - 内容: [概要]

...

### 調査範囲
- [チェック項目1]
- [チェック項目2]
- ...

### 関連ノート
- [[ノート1]]
- [[ノート2]]
```

注意事項:
- **最重要**: 1機能1ファイル、1概念1ファイルを厳守（まとめファイルは作成しない）
- 表面的な情報ではなく、実践的に使える詳細な内容を書く
- コード例は必ず含める
- 複数の情報源を参照する
- 公式ドキュメントを優先する
- 作成したファイルは必ずリンクする
- 各ファイルは独立して完結している必要がある
- 「○○まとめ.md」のような包括的なファイルは作成しない
