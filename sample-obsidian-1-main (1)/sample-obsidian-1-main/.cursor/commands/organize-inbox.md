# Inbox整理・振り分け支援

## Description
Inboxフォルダ内のファイルを適切なフォルダに振り分けます。

**重要**: 全てのファイルの中身を必ず読み込んで、内容を丁寧に分析・理解・抽象化してから振り分けます。

**実行原則**:
1. **必ずファイルの中身を読む** - ファイル名だけで判断しない
2. **内容を理解する** - 何について書かれているかを捉える
3. **抽象化する** - 本質的な概念や知識を抽出する
4. **適切に振り分ける** - 理解した内容に基づいて判定する

## 使用するツール
- `read_file`: ファイル内容を読み込む（必ず全内容を読み込む）
- `run_terminal_cmd`: ファイルの移動（mv コマンド）
- `list_dir`: フォルダ構造の確認

## Prompt
Inboxフォルダ内のファイルを内容に基づいて適切なフォルダに振り分けてください。

引数:
- ファイル名: 振り分け対象のファイル（ユーザーが指定）

実行内容:
1. **必ずファイルの中身を全内容読み込む**（read_file toolを使用）
   - ファイル名だけで判断しない
   - ファイル名が不明瞭でも内容から判断する
   - ファイルのサイズが大きくても必ず全内容を読む
2. **内容を理解して抽象化する**
   - ファイルの本質的な内容を捉える
   - 表面的な文字列ではなく、意味や意図を理解する
   - 重要な概念や知識を抽出する
   - 複数のトピックが含まれている場合は、それぞれを分析する
3. **内容から振り分け先を判定**
   - 抽象化した内容に基づいて判定
   - フォルダ間の使い分けガイドを参照
   - サブフォルダも考慮（04_Memory、03_Inputなど）
   - 代替案も検討する
4. **適切なフォルダへの振り分けを実行**
   - 判定理由を明確に記録
   - 一括で自動的に移動を実行
   - サブフォルダも自動的に指定
5. **移動後、必要に応じてメタデータを更新**
   - タグの追加・修正
   - リンクの追加
   - 関連ノートへのリンク追加

## 📊 フォルダ間の使い分けガイド

### Memory vs Input
- **04_Memory**: 3ヶ月後も参照する価値がある知識（長期記憶）
  - 自分の言葉で書かれている
  - 品質スコア7点以上
  - 例: 「Cursor Keyboard Shortcuts.md」
- **03_Input**: 今週〜今月使う資料（短期記憶）
  - 今まさに使っている情報
  - 1週間以上触っていない場合は移動を検討
  - 例: 「this-week-focus.md」

### Projects vs Areas
- **05_Output/Projects**: 期限ありのプロジェクト
  - 実行が決まっている
  - 具体的な仕様がある
  - 終わりがある
  - 例: 「SURVIBE AI 12月講座」
- **05_Output/Areas**: 継続的な活動（期限なし）
  - 継続的なエリア管理
  - 終わりがない
  - 例: 「YouTubeチャンネル運営」

### Daily vs Memory
- **02_Daily**: その日の記録（エピソード記憶）
  - 日付フォルダ形式（YYYY/YYYY-MM/YYYY-MM-DD/）
  - その日の出来事・気づき
  - 例: 「2025-01-13-Daily.md」
- **04_Memory**: 長期記憶として残す知識
  - 3ヶ月後も参照する価値がある
  - 例: 「Prompt Engineering Basics.md」

### Input vs Memory
- **03_Input**: 今週〜今月使う資料
  - 頻繁に見るチートシート
  - 進行中のタスクリスト
  - 例: 「reference-ai-tools.md」
- **04_Memory**: 長期記憶として残す知識
  - 体系的に整理された知識
  - 例: 「AI Tools Comparison.md」

## 📋 出力形式

振り分け完了後に以下の形式でサマリーを出力：

```markdown
## ファイル振り分け完了

### 移動したファイル一覧

| ファイル名 | 移動先フォルダ | 移動先サブフォルダ | 判定理由 |
|-----------|-------------|-----------------|---------|
| [ファイル名] | [フォルダ] | [サブフォルダ] | [理由] |

### 振り分け統計
- Memory: [件数]
- Input: [件数]
- Output/Projects: [件数]
- Output/Areas: [件数]
- Daily: [件数]
- Archive: [件数]
- その他: [件数]

### 合計
全 [合計件数] ファイルを整理しました。
```

手順:
1. **指定ファイルを必ず全内容を読み込む**（read_file tool使用）
   - ファイル名だけで判断しない
   - ファイル名が曖昧でも内容から判断する
   - ファイルが大きくても分割してでも全内容を読む
2. **内容を理解して抽象化する**
   - 「このファイルは何について書かれているか？」を理解する
   - 重要な概念や知識を抽出する
   - 複数のトピックが含まれている場合は、それぞれを分析する
   - ノートの本質的な目的を捉える
3. **抽象化した内容から振り分け先を判定**
   - フォルダ間の使い分けガイドを参照
   - サブフォルダも考慮（04_Memory、03_Inputなど）
   - Memory vs Input などの違いを理解して判定
   - 代替案も検討する
4. **判定結果を記録**
   - 内容要約を明確に記述
   - 判定理由を具体的に記録
   - サブフォルダも指定
5. **一括で自動的に移動を実行**
   - mv コマンドでファイルを移動
   - サブフォルダが存在しない場合は作成
   - 承認不要で自動実行
6. **04_Memory** の場合は、適切なサブフォルダ（AI/、Business/、Education/、Personal/、Technical/）に配置し、該当するMOCを更新
7. **02_Daily** の場合は、日付フォルダ形式（YYYY/YYYY-MM/YYYY-MM-DD/）に配置
8. **必要に応じてファイルの内容を更新**
   - タグの追加・修正
   - リンクの追加
   - 関連ノートへのリンク追加
9. **振り分け完了後にサマリーを出力**
   - 移動したファイルの一覧
   - 各ファイルの移動先
   - 振り分けの統計情報

注意事項:
- **最重要**: ファイル名だけで判断せず、必ずファイルの中身を全て読んでから振り分けを判断すること
- **必須**: read_file toolを使ってファイルの全内容を読み込むこと
- **必須**: ファイルの内容を理解して抽象化してから振り分け先を判定すること
- **必須**: ファイル名が不明瞭でも、内容から本質的なトピックを理解すること
- **必須**: ファイルが大きくても、分割してでも全内容を読むこと
- ファイル移動時はGit履歴を保持
- サブフォルダが存在しない場合は作成
- ファイル名の日付接頭辞は保持
- 移動後、必要に応じてメタデータ（タグ・リンク）を更新
- **04_Memory** への移動時は必ずサブフォルダを指定（AI/、Business/、Education/、Personal/、Technical/のいずれか）し、該当するMOCを更新
- **02_Daily** は日付フォルダ形式（YYYY/YYYY-MM/YYYY-MM-DD/）を厳守
- 判断に迷う場合は **99_Archive** に移動
- 複数のトピックが含まれている場合は、それぞれを分析して適切に扱う
- 表面的な文字列ではなく、ファイルの意味や意図を理解する

## 📝 プロンプト集管理ルール

### プロンプトの収集と整理
ログや実際の使用経験から、効果的だったプロンプトを収集してプロンプト集に追加します。

### プロンプト集のカテゴリ
**08_prompts/** フォルダに以下のカテゴリ別にプロンプトを保存:

#### 改善系（新規追加）
- コマンド改善系: Cursorコマンドファイルの改善プロンプト
- フォルダ管理系: フォルダ構成・命名規則に関するプロンプト
- システムメンテナンス系: ルールファイル更新のプロンプト

#### 既存カテゴリ
- 整理系: Inbox整理、Memory作成
- 生成系: Daily作成、ブログ記事作成
- レビュー系: ノート品質チェック
- 学習系: 技術調査

### プロンプトファイルの作成ルール
1. **カテゴリフォルダの確認**: 適切なカテゴリフォルダを選択（存在しない場合は作成）
2. **ファイル命名**: `YYYYMMDD_プロンプトタイトル.md` 形式
3. **必須項目**: 以下の情報を含める
   - **プロンプトタイトル**: 何に使うか明確に
   - **使用タイミング**: いつ使うか
   - **プロンプト本文**: 実際に使うプロンプト全文
   - **使用例**: 具体的な使用例
   - **期待される結果**: 何が得られるか
   - **注意事項**: 気をつけること
   - **関連プロンプト**: 関連する他のプロンプトへのリンク

### プロンプト収集の流れ
1. **ログからプロンプトを発見**
   - `.logs/` フォルダからよく使われているプロンプトを確認
   - 効果的だったプロンプトを特定
2. **カテゴリ判定**
   - プロンプトの用途を分析
   - 適切なカテゴリフォルダを選択
3. **プロンプトファイル作成**
   - 08_prompts/内の適切なカテゴリフォルダに作成
   - 必須項目を全て含めて記述
4. **README更新**
   - 08_prompts/README.md に新しいプロンプトを追加
   - カテゴリ一覧を更新

### よく使われるプロンプトパターン例

#### 改善系
